AV

    Detect, prevent, and remove malicious software (malware)

  • Types of Malware:
    • Viruses: Programs that replicate and spread between files or systems.
    • Trojans: Malicious software disguised as legitimate programs.
    • Worms: Spreads automatically across networks.
    • Spyware: Collects information without the user’s knowledge.
    • Ransomware: Encrypts data and demands payment for decryption.
    • Adware: Software that displays unwanted advertisements.
  • Types of AV Software:
    • Signature-Based Detection: Detects malware based on known virus signatures in the AV’s database.
    • Heuristic Detection: Analyzes the behavior of programs to detect new or unknown threats by their actions.
    • Cloud-Based AV: Uses cloud-based technology to offload malware detection and analysis, providing more up-to-date protection.
    • Real-Time Protection: Provides ongoing monitoring of system activities to block threats as soon as they are detected.
    • On-Demand Scanners: Allows users to scan their system manually at any time for potential threats.
  • How to Handle an Incident from AV:
    • When an AV detects a potential threat, it typically isolates the suspicious file to prevent it from spreading.
    • The Blue Team should investigate the alert, determine the scope of the attack, and perform forensic analysis to understand the attack vector.
    • If the incident is confirmed as a legitimate threat, steps should be taken to remove the malware and restore the affected systems from clean backups.
    • The incident should be logged, and lessons learned should be shared with the team to improve future detection and response.
  • Main Features of AV Software:
    • Real-Time Detection: Continuous monitoring of files and processes to detect and block threats as they occur.
    • File Scanning: Scanning files stored on devices for malware.
    • Quarantine: Isolating suspicious files to prevent them from harming the system.
    • Automatic Updates: AV software automatically updates virus definitions to detect new and emerging threats.
    • Web and Email Protection: Scanning links, email attachments, and websites for malicious content.

GDPR

GDPR – Key Points

  • Data Protection Law: A regulation from the EU that governs how personal data must be collected, processed, and stored.
  • Applies Globally: Affects any organization (inside or outside the EU) that handles the personal data of EU citizens.
  • Consent: Organizations must get clear and informed consent before collecting personal data.
  • Data Subject Rights:
    • Right to Access – Individuals can request access to their data.
    • Right to Rectification – Correct inaccurate data.
    • Right to Erasure (Right to be Forgotten).
    • Right to Data Portability – Transfer data between providers.
    • Right to Object – Refuse data processing in some cases.
  • Data Breach Notification: Must notify the authority within 72 hours of discovering a data breach.
  • Data Protection by Design: Privacy must be built into systems and processes from the start.
  • Data Protection Officer (DPO): Required for certain organizations to oversee GDPR compliance.
  • Heavy Penalties: Fines up to €20 million or 4% of global annual revenue, whichever is higher.
  • Accountability: Organizations must demonstrate compliance with GDPR principles.

Information Security Tickets

This involves tracking, prioritizing, and resolving security-related tickets and requests raised by colleagues to ensure a secure work environment.

Key Steps for Managing Security Tickets

  • 1. Ticket Creation:

    Colleagues raise security-related tickets (e.g., access requests, suspicious activity reports, system vulnerabilities). Tools like Jira, ServiceNow, or Zendesk may be used.

  • 2. Ticket Categorization:

    Classify tickets based on urgency and type (e.g., incident, request, vulnerability). This helps in assigning priority.

  • 3. Prioritization:

    Determine the priority of tickets. Critical security incidents (e.g., breaches or vulnerabilities) should be handled first, followed by less urgent requests.

  • 4. Investigation and Resolution:

    Analyze the issue in the ticket (e.g., malware alerts, access issues), take corrective action, and resolve it. This may involve collaborating with other teams (e.g., IT, development).

  • 5. Communication:

    Regularly update the ticket submitter on the status and resolution. Clear communication helps keep users informed and satisfied.

  • 6. Documentation:

    Document the process and steps taken to resolve the issue. This helps in future troubleshooting and compliance audits.

  • 7. Ticket Closure:

    Once the issue is resolved, close the ticket and mark it as completed. Ensure that the requester confirms the issue is fixed before closure.

  • 8. Review and Improvement:

    Review recurring issues or trends to improve security policies, practices, and systems. Identify areas where proactive measures can be introduced.

Examples of Common Information Security Tickets

  • Access Requests: Employees requesting access to secure systems or data.
  • Incident Reports: Colleagues reporting suspicious activities or potential security breaches.
  • Phishing Scams: Tickets reporting phishing emails or fraudulent activities.
  • System Vulnerabilities: Issues reported by vulnerability scanning tools or colleagues about weaknesses in the system.
  • Policy Violations: Tickets regarding the violation of company security policies (e.g., weak passwords or unauthorized software installations).

What is a SOC (Security Operations Center)?

A SOC is a team that monitors and responds to cybersecurity threats in real time.

Key Points

  • 24/7 Monitoring: Constant watch over systems and networks.
  • Threat Detection: Finds suspicious or harmful activity.
  • Incident Response: Acts fast to stop and fix attacks.
  • Log Analysis: Looks at system logs to understand what happened.
  • Tools: Uses SIEM, EDR, firewalls, antivirus, and DLP.
  • Team Roles:
    • Tier 1 – Handle alerts
    • Tier 2 – Investigate deeper
    • Tier 3 – Hunt threats
    • Manager – Lead the SOC
  • Supports Compliance: Helps meet security rules (e.g., ISO, GDPR).
  • Proactive Defense: Finds weak spots before attackers do.
> SOC Phases
  • 1. Preparation – Set up tools and team.
  • 2. Detection – Spot suspicious activity.
  • 3. Triage – Check if the alert is real.
  • 4. Investigation – Find out what happened.
  • 5. Containment – Stop the threat from spreading.
  • 6. Eradication – Remove the threat.

Information Security Controls

Information Security Controls are measures put in place to safeguard data, prevent unauthorized access, and protect systems from cyber threats.

>Information Security Controls Types
  • 1. Preventive Controls:

    These controls aim to prevent security incidents from happening. Example: Firewalls, strong password policies, and encryption.

  • 2. Detective Controls:

    These controls detect security incidents as they happen. Example: Intrusion Detection Systems (IDS), security cameras, and log monitoring.

  • 3. Corrective Controls:

    These controls correct security incidents after they occur. Example: Patching vulnerabilities, restoring data from backups, and applying security updates.

  • 4. Detective Controls:

    These controls help identify incidents in real time. Example: System audits, network traffic monitoring, and real-time security alerts.

  • 5. Recovery Controls:

    These controls help to recover from a security incident. Example: Data backup systems, disaster recovery plans, and business continuity planning.

  • 6. Compensatory Controls:

    Alternative controls implemented when primary controls cannot be used. Example: If a firewall fails, using an IDS as a backup detection mechanism.

Incident Response

Incident Response is the process of managing and responding to cybersecurity incidents, such as attacks or breaches.

Key Steps with Examples

  • 1. Preparation:

    Example: A company sets up an Incident Response Plan (IRP), deploys SIEM systems for monitoring, and trains its staff on how to react to common threats like phishing or malware.

  • 2. Detection:

    Example: A SOC team detects suspicious activity through their SIEM tool, which shows multiple failed login attempts from an unfamiliar IP address, indicating a possible brute force attack.

  • 3. Containment:

    Example: The team isolates the compromised system by disconnecting it from the network, preventing the attacker from moving laterally to other devices or systems.

  • 4. Eradication:

    Example: After identifying the malicious software, such as ransomware, the team removes the malware from all affected systems and patches the exploited vulnerability.

  • 5. Recovery:

    Example: The IT team restores data from secure backups and ensures all systems are clean before bringing them back online, verifying that the attack is fully neutralized.

  • 6. Lessons Learned:

    Example: After the incident, the team holds a debriefing session, analyzes what went well and what could have been handled better, and updates the Incident Response Plan to improve future responses.

Authentication vs Authorization
Definition

Authorization determines what actions or resources a user is allowed to access

Authentication is the process of verifying that a user is who they claim to be. There are three main types of authentication:

  • Something you know: Such as a password or the answer to a security question. These are sometimes called "knowledge factors".
  • Something you have: This is a physical object such as a mobile phone or security token. These are sometimes called "possession factors".
  • Something you are: For example, your biometrics or patterns of behavior. These are sometimes called "inherence factors".

Authorization involves verifying whether a user is allowed to do something, and is often combined with authentication for comprehensive access control.

How/Where to Detect

Authentication vulnerabilities can be detected during penetration testing, code reviews, and automated security scans. Tools like Burp Suite or OWASP ZAP can help test the authentication flow and identify weaknesses such as weak password policies, missing MFA, or insecure session management. It is important to also test for issues like session hijacking and improper access controls.

Mitigation
  • Use multi-factor authentication (MFA) to add an additional layer of security beyond just a password.
  • Implement rate-limiting and CAPTCHA on login forms to prevent brute-force attacks.
  • Encourage users to use unique, strong passwords for different sites or services.
  • Ensure proper session management by using secure cookies with the HttpOnly and Secure flags to protect session tokens.
  • Prevent session hijacking by using HTTPS for secure data transmission.
  • Fix logic flaws and poor coding practices in the authentication and authorization processes.
  • Improve password recovery functionality with proper validation and security measures to avoid exploitation.
  • Ensure that session durations are appropriately short to limit exposure.
  • Incorporate strong encryption methods to prevent Man-in-the-Middle (MitM) attacks.
  • Enforce the use of MFA and adequate encryption for sensitive data exchanges.
Types
  • Password-Based Authentication: The most common form, where users provide a username and password to authenticate.
  • Multi-Factor Authentication (MFA): Enhances security by requiring two or more forms of authentication (e.g., something you know, something you have, or something you are).
  • Token-Based Authentication: Relies on tokens (e.g., JWT, OAuth tokens) instead of passwords. Commonly used for API authentication and single sign-on (SSO) systems.
  • Biometric Authentication: Uses unique physiological traits (e.g., fingerprints, facial recognition) for authentication.
  • Certificate-Based Authentication: Uses digital certificates to authenticate users or systems in secure communications.
Example
POST /api/v1/login
{
  "username": "user1",
  "password": "securePassword123"
}
Response: 200 OK, { "token": "jwt-token" }

In this example, a user sends their username and password to authenticate and receives a token in response. This token can then be used for accessing protected resources.

HTTP Security Header

Authentication can be enhanced using headers such as Authorization (for token-based systems) and WWW-Authenticate for challenges in HTTP requests. Additionally, ensuring Strict-Transport-Security (HSTS) is enabled helps secure the transmission of sensitive authentication data.

API Testing
Definition

API Testing refers to the process of testing application programming interfaces (APIs) to ensure they function correctly and securely. This includes validating response accuracy, performance, and security vulnerabilities such as injection, authentication failures, and more.

How/Where to Detect

API testing is often done using tools like Postman, Burp Suite, or OWASP ZAP. It involves sending requests to API endpoints, analyzing responses, and testing for common vulnerabilities like input validation, authentication, and data exposure. Automated scripts or manual exploration are both commonly used methods.

Mitigation
  • Ensure proper input validation and sanitation to prevent injection attacks.
  • Implement strong authentication mechanisms such as OAuth, JWT, and API keys.
  • Use HTTPS for secure data transmission and protect data in transit.
  • Rate limit API requests to prevent abuse and denial-of-service attacks.
  • Utilize API gateways for additional protection and monitoring.
Types
  • Functional Testing: Ensures the API performs its intended functions correctly (e.g., checking if an API returns the correct data when a valid request is made).
  • Security Testing: Verifies that the API is secure, handling authentication, authorization, and sensitive data properly (e.g., testing for SQL injection, data leaks, and access control vulnerabilities).
  • Load Testing: Verifies the API's performance under various levels of load (e.g., checking how the API handles high traffic or stress).
  • Error Handling Testing: Tests how the API responds to invalid or erroneous inputs (e.g., incorrect parameters, unauthorized requests).
  • Boundary Testing: Tests the API by providing input at boundary or edge conditions (e.g., maximum length of input).
Example
POST /api/v1/login
{
  "username": "user1",
  "password": "pass1234"
}
Response: 200 OK, { "token": "jwt-token" }

If an API endpoint doesn't properly authenticate the user, it could lead to unauthorized access.

HTTP Security Header

Ensure APIs use security headers like Strict-Transport-Security, X-Content-Type-Options, and Content-Security-Policy to enhance security. Authorization headers should also be used to enforce proper authentication.

Address Resolution Protocol (ARP)

IP --> MAC within a local network.

ARP allows a device to find out what MAC address is associated with an IP address — which is essential for delivering packets at the data link layer.

How it works:

  • A device wants to send data to an IP (e.g., 192.168.1.5).
  • It broadcasts an ARP request on the network asking:
    "Who has IP 192.168.1.5? Tell me your MAC address."
  • The target device replies with its MAC address.
  • The sender stores this info in its ARP cache and sends the packet.

Base64 vs Base64URL Encoding

Base64 Encoding: Base64 is a binary-to-text encoding scheme that represents binary data in an ASCII string format by translating it into a radix-64 representation. It is commonly used in email encoding, data transmission, and file storage.

Buffer Overflow Attack

A buffer overflow attack occurs when an attacker sends more data to a buffer than it can handle, causing the excess data to overwrite adjacent memory. This can lead to unpredictable behavior, such as crashing the system or executing malicious code that grants the attacker unauthorized access to the system.

How It Works:

  • The attacker inputs data that exceeds the buffer size.
  • The excess data overwrites adjacent memory locations, potentially altering the program’s execution flow.
  • In some cases, the attacker can insert malicious code (e.g., shellcode) into the overflowed buffer, which is then executed by the system.

Goal of Attack:

  • Code Execution: Gain control of the vulnerable system by executing arbitrary code.
  • Privilege Escalation: Elevate the attacker's privileges to gain unauthorized access to sensitive areas of the system.

Prevention Techniques:

  • Bounds Checking: Ensure that buffers cannot overflow by validating the size of input data.
  • Stack Canaries: Use security features that detect buffer overflows before critical data is overwritten.
  • Data Execution Prevention (DEP): Prevent the execution of code from certain regions of memory.
  • Address Space Layout Randomization (ASLR): Randomize memory addresses to make it more difficult for attackers to predict memory locations.
Brute Force Attack
Definition

A brute force attack is a trial-and-error method used by attackers to guess passwords, encryption keys, or other sensitive data by systematically testing every possible combination until the correct one is found.

How/Where to Detect

Brute force attacks can be detected by monitoring failed login attempts, especially when the number of failures exceeds a certain threshold. Other indicators include high traffic volumes directed at authentication pages or systems, which might suggest repeated login attempts.

Mitigation
  • Implement account lockout policies after a certain number of failed login attempts.
  • Use CAPTCHA or reCAPTCHA on login forms to deter automated attacks.
  • Enforce strong password policies (e.g., requiring a mix of letters, numbers, and symbols).
  • Use multi-factor authentication (MFA) to add an extra layer of protection.
  • Implement rate limiting or delay login attempts to slow down brute force attacks.
Types
  • **Simple Brute Force**: Trying every possible combination of characters until the correct one is found.
  • **Dictionary Attack**: Using a list of commonly used passwords (dictionary words) to guess the correct password.
  • **Hybrid Attack**: A combination of brute force and dictionary attacks, where common passwords are tested with variations (e.g., adding numbers or special characters).
  • **Credential Stuffing**: Using stolen username and password combinations from previous breaches to attempt login on other services.
Example

An attacker might attempt to access an account on a website by using a script that tries every possible combination of characters for the password field, systematically entering each combination until they find the correct one. If the attacker uses a dictionary of common passwords, they might only need to try a few hundred or thousand combinations before succeeding.

HTTP Security Header

HTTP security headers can help mitigate brute force attacks by adding additional layers of protection. For example:

  • X-Frame-Options: Prevents clickjacking attacks that could be used in conjunction with brute force attempts to trick users into submitting their credentials.
  • Strict-Transport-Security (HSTS): Forces the use of HTTPS to secure login credentials from being intercepted during brute force attacks.
  • X-Content-Type-Options: Prevents MIME type sniffing, which could help mitigate some types of attacks where brute force attempts target vulnerable content types.
  • Content-Security-Policy (CSP): Reduces the risk of XSS attacks, which could assist an attacker trying to exploit vulnerabilities to facilitate brute force attacks.

CVE, CWE, CVSS, and SLA

CVE (Common Vulnerabilities and Exposures): A unique identifier for publicly known vulnerabilities. Example: CVE-2021-34527 refers to the "PrintNightmare" vulnerability in Windows.

CWE (Common Weakness Enumeration): A catalog of software weaknesses that lead to vulnerabilities. Example: CWE-89 is "SQL Injection," a weakness in handling user input safely in SQL queries.

CVSS (Common Vulnerability Scoring System): A score to measure the severity of a vulnerability. Example: A CVE might have a CVSS score of 9.8, indicating it is critical (e.g., remote code execution).

How CVSS is Calculated: CVSS is calculated based on several factors, which are grouped into three categories:

  • Base Score: Measures the intrinsic severity of the vulnerability (e.g., impact on confidentiality, integrity, and availability).
  • Temporal Score: Measures the current exploitability of the vulnerability (e.g., known exploits, fix availability).
  • Environmental Score: Measures the specific impact to the organization based on their environment.

Calculation Formula: CVSS Base Score = (Impact + Exploitability) * Temporal * Environmental factors. CVSS is generally calculated using a formula that factors in things like how easily the vulnerability can be exploited (Exploitability), the impact it has on the system (Impact), and the availability of mitigations or fixes.

SLA (Service Level Agreement): Defines the expected time for resolving a vulnerability. Example: "Fix the vulnerability within 48 hours" or "Resolution in next monthly patch cycle."

HTTP Request Smuggling
Definition

HTTP Request Smuggling occurs when an attacker exploits discrepancies between how different components of a system (e.g., web servers, proxies) interpret HTTP requests. This can lead to unexpected behavior, such as bypassing security mechanisms, unauthorized access, or response splitting.

How/Where to Detect

Detection typically involves analyzing the interaction between different layers (e.g., proxy, web server, backend) and how they process HTTP requests. This can be detected by sending specially crafted HTTP requests and observing abnormal server responses.

Mitigation
  • Ensure proper configuration and synchronization of proxies and backend servers to handle HTTP requests uniformly.
  • Validate and sanitize HTTP request headers to ensure no ambiguous request formats can bypass the system.
  • Use a single HTTP parser for all components to avoid discrepancies.
  • Implement security mechanisms like input validation, rate limiting, and logging for unusual requests.
Types
  • Request Smuggling between front-end proxies and back-end servers.
  • HTTP request splitting attacks.
Example

Example of an HTTP Request Smuggling attack:

    POST / HTTP/1.1
    Host: victim.com
    Content-Length: 13
    Transfer-Encoding: chunked
    X-Forwarded-For: 

    0\r\n\r\nPOST /evil HTTP/1.1
    Host: victim.com

In this example, the attacker sends a request that is interpreted differently by the front-end proxy and the back-end server, potentially allowing malicious requests to bypass security measures.

No specific HTTP header for HTTP Request Smuggling prevention, but headers like Strict-Transport-Security (HSTS) can be used to ensure secure connections and mitigate attacks on the transport layer.

ISO 27001

  • What? ISO 27001 is an international standard that defines the requirements for an Information Security Management System (ISMS). It helps helps organizations secure their information systematically and effectively — protecting customer data, internal processes, and critical systems.
  • Information Security Management System (ISMS) is a set of policies, procedures, and controls that organizations put in place to systematically manage and protect their information assets.
  • Risk-based Auditing: audit plan designed to address management’s high-priority risks. It allows internal auditors to respond to the risks and provide insights to management to solve risks timely.
  • Types of Audit Risks: Control Risks (The risk that security measures won't work as expected); Detection Risk (The risk that a problem or breach won't be noticed in time); Inherent Risk (The risk that existing controls will fail to prevent or detect issues);
  • Risk Assessment Report (RAR) - A standard report that includes the outcomes of a risk assessment, detailing the findings and action plan based on the identified risks in an organization.
  • Statement of Applicability - A document that identifies which controls from ISO 27001 are applicable to the organization and which are not, along with the justification for these decisions.
  • Risk Treatment Plan - A plan that outlines how identified risks will be treated (e.g., mitigating, accepting, transferring, or avoiding), including actions and timelines.
Steps of ISO 27001 risk assessment
  • Define the risk assessment method - Choose an appropriate approach or methodology for assessing risks (e.g., qualitative or quantitative).
  • Compile the list of information assets - Identify and list all critical information assets that need protection (e.g., data, software, systems).
  • Identify the threats and vulnerabilities - Identify potential threats (e.g., cyber-attacks) and vulnerabilities (e.g., outdated software) that could impact assets.
  • Evaluate risks - Assess the likelihood and potential impact of each identified risk.
  • Mitigate risks - Apply controls or actions to reduce or eliminate risks to an acceptable level.
  • Develop risk reports - Document the results of the risk assessment, including identified risks and mitigation plans.
  • Review, monitor, and audit - Regularly review, monitor, and audit the risk management process to ensure it remains effective and up to date.
Security Policies
  • Protect Confidentiality - Ensure sensitive information is accessible only to authorized individuals.
  • Ensure Integrity - Safeguard information from being tampered with, altered, or destroyed.
  • Maintain Availability - Ensure that critical data and systems are accessible and usable when needed.
  • Compliance - Ensure the organization meets legal, regulatory, and contractual security requirements.
  • Risk Management - Identify, assess, and mitigate security risks to reduce vulnerabilities.
  • Employee Awareness - Educate employees on security risks and best practices for safeguarding company information.
  • Incident Response - Define procedures for responding to security breaches or threats effectively and quickly.

Definition

Insecure deserialization occurs when an attacker manipulates serialized data to execute arbitrary code, gain unauthorized access, or perform other malicious activities. It is common in applications that accept serialized objects from untrusted sources without proper validation or integrity checks.

How/Where to Detect

Insecure deserialization can be detected by analyzing how serialized data is handled, especially when it's coming from untrusted sources. Look for unprotected deserialization routines in the application code and monitor for unusual deserialization requests or errors in logs.

Mitigation

  • Use strong encryption and signing of serialized data to prevent tampering.
  • Avoid deserializing objects from untrusted sources whenever possible.
  • Implement access controls to restrict access to sensitive deserialization routines.
  • Use safe deserialization libraries that check the integrity of the data.
Types
  • Object injection attacks
  • Remote code execution (RCE) through deserialization
  • Privilege escalation by tampering with serialized data

Indicators of Compromise (IOCs)

Indicators of Compromise (IOCs) are signs that a system or network has been compromised by a cyberattack. Monitoring these indicators helps organizations detect and respond to security breaches quickly.

Common IOCs to Monitor:

  • Unusual Network Traffic: Unexpected spikes or patterns in traffic, often indicating data exfiltration or malware communication.
  • Unauthorized Logins: Multiple failed login attempts, or logins from unfamiliar locations or devices, can indicate a brute-force attack.
  • Changes in File Integrity: Unexpected modifications to critical system files, including system configurations or executable files.
  • Suspicious Processes: Malicious or unauthorized processes running in the background, which can indicate the presence of malware.
  • New or Unknown User Accounts: Creation of new user accounts without authorization, often used to escalate privileges.
  • Increased Resource Usage: Unexplained high CPU or memory usage, which could be caused by a malware infection or resource-intensive attacks.
  • Unusual Outbound Communications: Unexpected outbound communications, especially to foreign or unrecognized IP addresses, can signal a data breach or exfiltration.
  • Presence of Malware Signatures: Detection of known malicious files, URLs, or hash values on the system.

Importance of Monitoring IOCs:

  • Helps in early detection of a breach or attack.
  • Provides evidence for investigating and mitigating security incidents.
  • Improves the overall security posture by identifying weaknesses in the network.
J
JWT (JSON Web Token)
Example

Imagine a user logs in to a web application, and the server issues a JWT to the client after successful authentication. This token contains the user's ID and roles. In future requests, the client sends this token in the `Authorization` header to authenticate and authorize actions.

    Authorization: Bearer
Definition

JSON Web Token (JWT) is a URL-safe token format for securely transmitting information between parties as a JSON object. It is commonly used for authentication and authorization purposes in web applications.

Mitigation
  • Tokens should have an expiration date to limit their lifespan.
  • Avoid sending tokens in URL parameters to prevent interception (use headers instead).
  • Enable the issuing server to revoke tokens (e.g., during logout) to prevent unauthorized use.
JWT Structure
    Header.Payload.Signature
Header

The header contains metadata such as the algorithm used for signing the token (e.g., HMAC, RSA). The header is base64-url encoded.

    {"alg": "HS256", "typ": "JWT"}
Payload

The payload contains the user data and metadata, such as the user’s ID, name, or roles.

    {"sub": "1234567890", "name": "John Doe", "email": "___", "iat": 1516239022}

iat: *Issued At* — timestamp when the JWT was issued.

Signature

The signature is used to verify the integrity and authenticity of the token. It is created by signing the base64-encoded header and payload with a secret key or private key (e.g., HMAC SHA-256).

    Signature = HMACSHA256(base64UrlEncode(header) + "." + base64UrlEncode(payload), secretKey)
Attacks
  • JWT Signature Modification: If the "alg" is set to "none", the signature can be removed, allowing an attacker to forge a token and impersonate a user with elevated privileges.
  • Accepting Tokens with No Signature: Some servers might accept tokens with an invalid signature or none at all, which allows attackers to modify or create their own tokens.
  • Brute Forcing: Weak or misconfigured JWT secrets can be exploited using brute force tools like hashcat to guess the secret key.
  • JWT Header Parameter Injection: Attackers can inject malicious parameters (e.g., `kid`, `jku`) into the JWT header, leading to vulnerabilities like path traversal or remote code execution.
JWT vs JWS vs JWE
  • JWS (JSON Web Signature): A JWT with a digital signature that ensures the integrity and authenticity of the data.
  • JWE (JSON Web Encryption): A JWT that is encrypted to ensure confidentiality, protecting the data from unauthorized access.

MITM
Definition

Attacker intercepts and potentially alters communication between two parties who believe they are directly communicating with each other.

How/Where to Detect
  • Check for unusual behavior in network traffic such as unexpected changes in IP addresses.
  • Monitor for **SSL/TLS** errors or expired certificates that may indicate tampering.
  • Analyze network traffic for unexpected delays or discrepancies in encryption.
  • Inspect HTTPS connections for **mixed content** issues or warning signs that the certificate chain has been compromised.
Mitigation
  • Use **SSL/TLS encryption** for all sensitive data transmission to ensure data is protected from interception.
  • Implement **Perfect Forward Secrecy (PFS)** to ensure that encryption keys are not reused across sessions.
  • Employ **HSTS (HTTP Strict Transport Security)** to enforce the use of HTTPS and prevent downgrade attacks.
  • Use **Multi-Factor Authentication (MFA)** to ensure that even if credentials are intercepted, access is not easily gained.
  • Regularly check and update SSL certificates, ensuring they are from trusted certificate authorities.
Types
  • SSL Stripping: The attacker downgrades a secure HTTPS connection to an unencrypted HTTP connection.
  • Session Hijacking: The attacker steals an active session token, allowing them to impersonate a legitimate user.
  • DNS Spoofing: The attacker redirects the victim’s network traffic to a malicious server by manipulating DNS responses.
  • Packet Injection: The attacker injects malicious packets into the communication stream, potentially altering the data being transmitted.
Example

An example of a MITM attack is when an attacker sets up a fake Wi-Fi hotspot (e.g., "Free Public Wi-Fi") in a public place and intercepts the network traffic of users who connect to it, potentially capturing sensitive data like login credentials or credit card details.

HTTP Security Header
  • Strict-Transport-Security (HSTS): A security header that forces browsers to only communicate with the server using HTTPS, mitigating certain types of MITM attacks.
  • Content-Security-Policy (CSP): Helps prevent malicious content from being injected into web pages, which could be used in a MITM attack to deliver malicious payloads.
  • X-Content-Type-Options: Ensures that browsers strictly interpret content as its declared type, preventing attackers from injecting malicious content in unexpected formats.

OSI Model: framework that standardizes the functions of a computer network into seven layers, from physical data transmission to application-level communication. It helps in understanding how different network protocols interact and how data is transmitted across networks.

Layer Description Common Attacks Protocols
7. Application Layer Provides network services to end-user applications (e.g., HTTP, FTP, DNS).
  • XSS (Cross-Site Scripting): Malicious scripts injected into web applications.
  • SQL Injection: Exploiting vulnerabilities in SQL queries to manipulate databases.
  • DoS (Denial of Service): Overloading web or application servers to make them unavailable.
  • HTTP, FTP, DNS, SMTP, POP3, IMAP
6. Presentation Layer Translates data between the application and transport layers (e.g., data encoding, encryption).
  • Man-in-the-Middle (MitM): Intercepting and altering data between the client and server.
  • SSL Stripping: Downgrading a secure HTTPS connection to HTTP for eavesdropping.

5. Session Layer Manages sessions and connections between applications (e.g., establishing and terminating connections).
  • Session Hijacking: Stealing a valid session token to impersonate the user.
  • Session Fixation: Attacker sets a session ID and uses it to hijack a session later.
  • NetBIOS, RPC, SMB, PPTP
4. Transport Layer Ensures reliable data transfer and error handling (e.g., TCP, UDP).
  • TCP Spoofing: Sending fake TCP packets to manipulate connections.
  • UDP Flood: Sending large volumes of UDP packets to overload the network.
  • Man-in-the-Middle (MitM): Modifying data in transit between the client and server.

  • IP Spoofing: Falsifying the source IP address of packets to hide the origin of the attack.
  • Routing Attacks: Manipulating routing tables or protocols to redirect traffic (e.g., BGP hijacking).
  • DoS (Denial of Service): Attacks like Ping of Death that flood the network with malformed packets.
  • IP, ICMP, ARP, OSPF, BGP

  • Ethernet, Wi-Fi (IEEE 802.11), PPP, ARP, Frame Relay
1. Physical Layer Transmits raw bitstreams over physical media like cables or wireless signals.
  • Wiretapping: Intercepting physical signals to capture sensitive information.

Pharming (fake website)

Pharming is a type of cyber attack where a user is directed to a fraudulent website that appears to be legitimate, typically by manipulating the DNS (Domain Name System) or altering host files.

How It Works:

  • The attacker compromises a DNS server or the user's local machine.
  • The victim tries to visit a legitimate site (e.g., their bank’s website).
  • Instead of the real site, they are redirected to a fake site that looks identical.
  • The attacker captures sensitive data entered by the victim.

Key Characteristics:

  • Invisible to users — the redirection happens without their knowledge.
  • Highly effective — users trust the fraudulent site because it looks legitimate.

Session Fixation is a type of Session Hijacking attack where:

SAML
Definition

SAML is a widely used standard for Single Sign-On (SSO) that allows users to authenticate once and access multiple services. However, like any protocol, it has vulnerabilities that can be exploited by attackers if not properly secured.

How/Where to Detect

SAML-based authentication is typically detected through logs, network traffic analysis, and web application security monitoring. Look for SAML tokens or assertions being transmitted in HTTP headers.

SAML Vulnerabilities

Common SAML Vulnerabilities:

  • SAML Assertion Manipulation: Attackers can manipulate the SAML assertion (authentication data) to gain unauthorized access by altering or injecting fake assertion data.
  • Assertion Replay: Attackers can intercept and replay valid SAML assertions to impersonate users and gain unauthorized access.
  • Weak or Missing Signature Validation: If the SAML assertion is not properly signed or validated, attackers can forge or modify assertions, potentially allowing them to impersonate legitimate users.
  • XML Signature Wrapping (XSW): Attackers can tamper with the XML structure of the SAML message, bypassing signature verification and altering the data in the request.
  • Open Redirects: A vulnerability where a malicious attacker tricks the service provider into redirecting a user to a malicious URL during the SAML process, potentially leading to phishing attacks.
  • SAML Token Hijacking: If a SAML token is intercepted or leaked, attackers can reuse it to impersonate users and gain unauthorized access to services.
  • Misconfigured Identity Providers (IdP): Incorrect configurations or weak security practices on the IdP side (e.g., default or weak passwords, lack of MFA) can lead to attackers gaining access to sensitive resources.

Mitigating SAML Vulnerabilities:

  • Enforce Strong Signature Validation: Always validate SAML assertions and responses with the appropriate signatures to ensure their integrity and authenticity.
  • Use Encrypted Assertions: Encrypt SAML assertions to protect sensitive information and prevent unauthorized access during transmission.
  • Implement MFA: Use Multi-Factor Authentication (MFA) for additional protection during SAML-based logins to reduce the risk of token hijacking or exploitation.
  • Limit Assertion Lifespan: Set a short expiration time for SAML tokens to minimize the risk of token replay attacks.
  • Regular Security Audits: Regularly audit and review your SAML implementation to ensure it adheres to security best practices and is resistant to common vulnerabilities.
Types
  • Single Sign-On (SSO): Allows users to authenticate once and access multiple systems without re-authenticating.
  • Federated Identity Management: Enables identity sharing across different domains or organizations.
Example

A typical example is when an employee logs in to a corporate network (using SAML) and is automatically granted access to other enterprise applications like email, HR systems, and cloud services without having to log in again.

HTTP Security Header

The SAML assertions are typically transmitted in the HTTP headers as part of the authentication process. It's important to secure these headers through proper encryption and validation to avoid interception and misuse.

  • SOAR (Security Orchestration, Automation, and Response): Automates incident response workflows by integrating different security tools and processes.
  • IDS (Intrusion Detection System): Monitors network or system activities for malicious activities or policy violations (detects but does not block).
  • IPS (Intrusion Prevention System): Like IDS but actively blocks or prevents detected threats in real-time.
  • EDR (Endpoint Detection and Response): Monitors, detects, and responds to threats specifically on endpoints like laptops and servers.

What is a SOC?

A SOC is a centralized team or facility responsible for continuously monitoring, detecting, analyzing, and responding to cybersecurity incidents within an organization.

How to Build a SOC

Step Description Example
Define Scope and Objectives Decide what the SOC will protect and its goals. Protect networks, cloud, and endpoints.
Build the Team Hire analysts, incident responders, SOC managers. Recruit 24/7 monitoring analysts.
Choose the Right Tools Select SIEM, EDR, IDS/IPS, threat intelligence tools. Use Splunk (SIEM) and CrowdStrike (EDR).
Develop Processes and Playbooks Create procedures for detection and response. Develop phishing attack playbooks.
Set Up Monitoring Infrastructure Centralize log collection and monitoring. Collect logs from firewalls, servers, endpoints.
Continuous Improvement Review SOC performance and update strategies. Run SOC maturity assessments.

Importance of SOC for Organizations

  • Early Threat Detection: Detect threats before they escalate.
  • Rapid Incident Response: Minimize damage and downtime.
  • Regulatory Compliance: Meet standards like GDPR, HIPAA, ISO 27001.
  • Business Continuity: Keep operations running securely.
  • Risk Reduction: Proactively manage and lower cybersecurity risks.

Examples of SOC Activities

  • 24/7 Threat Monitoring: Monitoring SIEM alerts around the clock.
  • Incident Response: Quickly isolating a malware infection.
  • Threat Hunting: Proactively finding hidden threats.
  • Security Reporting: Providing security status reports to leadership.

TCP/IP Internet Layer

The TCP/IP Internet Layer is responsible for addressing, routing, and forwarding packets between devices across networks. It ensures that data can travel from the source to the destination, even if they are on different networks. This layer focuses on the logical addressing and routing, as well as error handling and diagnostics.

Key Functions:

  • Routing: Determines the path that data packets take from source to destination across different networks.
  • Addressing: Uses logical IP addresses to uniquely identify devices on a network.
  • Packet Forwarding: Forwards data packets across network boundaries, ensuring they reach their correct destination.
  • Error Handling: Provides error reporting through protocols like ICMP.

Key Protocols:

  • IP (Internet Protocol): Routes packets based on IP addresses.
  • ICMP (Internet Control Message Protocol): Handles error messages and diagnostics.
  • ARP (Address Resolution Protocol): Resolves IP addresses to MAC addresses within local networks.
  • RARP (Reverse Address Resolution Protocol): Resolves MAC addresses to IP addresses (used less frequently today).

VPN (Virtual Private Network)

VPN and Types

A VPN (Virtual Private Network) is a service that creates a secure, encrypted connection over the internet between the user and a remote server. It helps protect data from hackers, provides privacy, and can bypass geographical restrictions.

Types of VPNs:

  • Remote Access VPN: Connects individual users to a private network remotely (e.g., employees accessing a company network).
    Example: Cisco AnyConnect, OpenVPN.
  • Site-to-Site VPN: Connects entire networks to each other (e.g., linking two branch offices).
    Example: MPLS VPN, IPsec VPN.
  • SSL VPN (Secure Sockets Layer VPN): Uses SSL/TLS protocols for secure remote access via a web browser.
    Example: OpenVPN (with SSL/TLS encryption).
  • IPsec VPN (Internet Protocol Security VPN): Provides encrypted communication at the IP layer for secure communication between two endpoints.
    Example: IPSec-based VPNs for site-to-site connections.
  • L2TP (Layer 2 Tunneling Protocol) VPN: Combines L2TP and IPsec for encryption, providing secure tunneling.
    Example: L2TP/IPsec for remote access.
  • PPTP (Point-to-Point Tunneling Protocol): One of the oldest VPN protocols, often considered less secure.
    Example: PPTP connections in legacy systems.

Web Cache Deception vs Web Cache Poisoning

Web Cache Attacks

1. Web Cache Deception

Web Cache Deception involves finding a dynamic page that can be accessed via a URL format that the web cache mistakenly treats as cacheable.
Example: Accessing /transactions via /transactions.jpg.

Attack Flow:

  • An attacker tricks a victim into visiting a crafted, cacheable URL.
  • The victim’s private information is stored in the cache under that URL.
  • The attacker loads the same URL and retrieves the victim’s data from the cache.

2. Web Cache Poisoning

Web Cache Poisoning involves finding an input that changes the response in an exploitable way, without affecting the cache key used by the server.

Attack Flow:

  • An attacker sends a request with a malicious payload.
  • The server responds with a modified response, but the cache stores this response under a "normal" cache key.
  • Other users receive the poisoned (malicious) response when accessing the same page.

A Zero-Day Vulnerability is a security flaw that is unknown to the vendor and has no fix available when discovered. Attackers exploit this vulnerability before the vendor can address it.
