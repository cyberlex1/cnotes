SQLi

Parameterized queries are SQL queries where user inputs are passed as parameters rather than being directly embedded into the query string. This prevents SQL injection by treating user input as data, not executable code. It ensures the query structure remains safe and intact.

SQLi Type Description Mitigation How to Detect Examples
In-Band SQL Injection attacker retrieves data directly from the error messages or query results. - Use parameterized queries .
- Disable detailed error messages in production.		 - Unexpected behavior or errors in the application.
- Form fields or URL parameters.		 SELECT * FROM users WHERE username = 'admin' AND password = 'password';
Payload: admin' OR 1=1 --

- Implement error handling that does not reveal details.		 - Detect by noticing delays or changes in application behavior (e.g., different response times based on valid/invalid input). SELECT * FROM users WHERE username = 'admin' AND password = 'password';

Union-based SQL Injection UNION operator to combine the results of multiple queries, allowing data extraction. - Use parameterized queries.
- Restrict database privileges for users.
- Implement proper input validation.		 - Responses from the application that include unexpected data.
- Tools like Burp Suite or SQLMap can automate detection.		 SELECT id, name FROM users WHERE username = 'admin' UNION SELECT username, password FROM admins;
Time-based Blind SQL Injection The attacker can infer information based on the time it takes the application to respond, using SLEEP() or similar functions to delay responses. - Use parameterized queries.
- Implement proper error handling and timeouts for database operations.		 - Detect by measuring response times for queries that include SLEEP() or other time delays.
- Compare response time differences between queries.		 SELECT * FROM users WHERE username = 'admin' AND password = 'password' AND IF(1=1, SLEEP(5), 0);
Out-of-Band SQL Injection The attacker uses an external server to retrieve data through the database, leveraging functions like LOAD_FILE or xp_cmdshell. - Use firewalling and network restrictions.
- Disable dangerous database functions like xp_cmdshell.		 - Detect by monitoring network traffic for unusual outbound connections.
- Look for abnormal data requests or unexpected DNS lookups.		 SELECT LOAD_FILE('/etc/passwd');
SELECT * FROM users WHERE id = 1; EXEC xp_cmdshell('nslookup example.com');
Second-Order SQL Injection The attacker injects malicious SQL into an application input field, but the query is not executed immediately, allowing time for the attacker to later trigger the SQLi. - Use parameterized queries.
- Regularly validate and sanitize inputs.
- Implement strict session and state management.		 - Detect by monitoring the application for suspicious sequences or patterns in multiple queries that eventually trigger the injection. Step 1: INSERT INTO users (username, password) VALUES ('admin', 'password123');
Step 2: SELECT * FROM users WHERE username = 'admin' AND password = 'password123';
