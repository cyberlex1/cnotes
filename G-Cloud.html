üîê Security Command Center (SCC)
What is it?
Google Cloud Security Command Center (SCC) is a security management platform that provides visibility, threat detection, vulnerability management, and compliance monitoring for Google Cloud resources. It helps detect threats, identify vulnerabilities, and automate responses, enhancing cloud security.
How to detect and respond to threats with SCC

SCC detects threats and vulnerabilities. It includes:

  • Event Threat Detection: Detects known patterns (e.g., brute force, malware downloads).
  • Cloud Anomaly Detection: Uses machine learning to find unusual behaviors (e.g., leaked credentials, spikes in traffic).

Other detectors:

  • Security Health Analytics: Finds misconfigurations based on best practices (CIS, NIST).
  • Qualys Cloud Security for SCC: Assesses OS-level and app vulnerabilities.
  • Cloud DLP: Detects sensitive data (e.g., PII, PCI) across GCP services.
  • Cloud Security Scanner: Scans for vulnerabilities in web apps (e.g., XSS, outdated libraries).
Response Process
  • 1. Activate SCC: Set up SCC for your projects or entire organization. Ensure proper IAM roles are configured.
  • 2. Enable Threat Detectors: Turn on detectors like Event Threat Detection, Cloud Anomaly Detection, and Qualys Cloud Security.
  • 3. Monitoring Findings: SCC scans and generates findings based on active detectors. These include severity and the Attack Exposure Score (AE Score).
  • 4. Automated Response with Cloud Functions: Findings trigger automated responses like isolating a VM or removing risky firewall rules using Cloud Functions via Pub/Sub.
  • 5. Alert Notifications: SCC sends alerts through email, Slack, or SIEM systems to notify the security team of critical findings.
  • 6. Incident Tracking: Log and track incidents in systems like Jira or ServiceNow for analysis and prioritization.
  • 7. Remediation: Apply fixes, remove vulnerabilities, and update security policies regularly based on findings.
  • 8. Compliance and Reporting: Ensure compliance with standards like CIS, NIST, and ISO 27001. Generate reports and export findings for analysis in BigQuery or Looker.

Examples

SCC detects a VM with SSH open to the internet
  • 1. SCC generates a critical finding: OPEN_FIREWALL.
  • 2. Finding sent to Pub/Sub.
  • 3. Cloud Function removes the risky firewall rule or isolates the VM.
  • 4. Notification sent to team (email, Slack, SIEM).
  • 5. Team analyzes, documents the issue, and adjusts policy.

üîê Identity and Access Management (IAM)

IAM in GCP

Controls who (identity) can do what (roles) on which resource in GCP

Has the principle of Least Privilege (*nobody should have more permissions than they need*) and Role-Based Access Control (RBAC) is a method of managing access to resources based on the roles assigned to users, groups, or service accounts.

How it works

  • Authentication: GCP checks who (user login, service account key, etc.) you are and then gives you authorization by checking what IAM policies are attached to a resource and checks what are the roles assigned to you.

who - Identity

  • Users: google accounts (`user@gmail.com`)
  • Service Accounts (svc account): Represent non-human entities: applications, virtual machines (VMs), etc. that need to authenticate and interact with Google Cloud resources programmatically.
  • Groups: Collections of users (`engineers@domain.com`)
  • External Identities: Identities from external identity providers (e.g., Azure AD, custom identity providers)
  • Principals: Abstract representation of entities (`allUsers`, `allAuthenticatedUsers`, custom service account)

The recommendation is always to work with groups where possible as individual users might leave the company, change teams, etc.

what - Access

Roles = 1 or more permissions. Roles are attached to identities. They define what type of actions an identity can do.

A user can have multiple roles for a single resource. When a user has multiple roles, the effective permissions are the union of the permissions granted by each role.

Types of Roles

  • Primitive Roles:
    • Owner: Full access to all resources, including managing IAM policies.
    • Editor: Can edit resources but cannot manage IAM policies.
    • Viewer: Read-only access to resources.
  • Predefined Roles: Contain permissions for specific services and tasks.
    • Storage Admin: Full control over Cloud Storage resources. `roles/storage.admin`
    • Compute Admin: Full control over Compute Engine resources. `roles/compute.instanceAdmin`
    • BigQuery Data Editor: Full access to BigQuery datasets and tables `roles/bigquery.dataEditor`
    • Kubernetes Engine Cluster Viewer: Read-only access to Kubernetes Engine clusters `roles/container.clusterViewer`
  • Custom Roles: Use when predefined roles do not precisely match the access needs of your organization or project.

E.g. Compute Admin role will give you the permissions to create virtual machines, networks, etc. `roles/compute.admin`

where - Management

Resources or IAM Objects are accessed by identities based on roles assigned to those identities.

GCP resources are organized in a strict resource hierarchy. Every resource must have a project => projects creation is mandatory.

IAM Policy

IAM Policy = list of rules called bindings

  • Each binding connects 1 or more identities (domain, user, group, svc account) to a specific role (contains permissions) on a specific resource (project, bucket, VM, etc.)
  • Policies can be defined at resource level, project level, folder level or organization level, depending on the type of resource.
  • IAM policies are inherited from parent resources to child resources in GCP.
    • E.g. if you set an IAM policy at the project level, all resources within that project inherit that policy unless explicitly overridden.
        {
          "bindings": [
            {
              "role": "roles/storage.objectViewer",
              "members": [
                "user:alice@example.com",
                "group:dev-team@example.com"
              ]
            },
            {
              "role": "roles/compute.admin",
              "members": [
                "serviceAccount:my-service-account@project.iam.gserviceaccount.com"
              ]
            }
          ]
        }
  • roles/storage.objectViewer: Grants read-only access to objects in Cloud Storage. Users can view and download objects but cannot create, modify, or delete them.
  • roles/compute.admin: Grants full administrative access to Compute Engine resources, including the ability to create, modify, and delete VMs, networks, and disks.

IAM Conditions

  • Make IAM role bindings more granular by adding "if..." rules to permissions.
  • Instead of simply saying "User X has Role Y", you can say "User X has Role Y _only if_ Condition Z is true."
Type Example
Resource-based Only access objects in a specific Cloud Storage bucket path.
Request-based Only allow access from a certain IP address.
Time-based Only allow access before a certain date (e.g., temporary access).

üîê Cloud Armor

Cloud Armor - Protecting Applications on Google Cloud Platform

Cloud Armor is a security service provided by Google Cloud Platform (GCP) that helps protect applications from a variety of threats, particularly those from malicious web traffic. Its primary role is to safeguard applications deployed in GCP by filtering and blocking malicious requests before they reach the application.

Key Features and Role of Cloud Armor:

  • DDoS Protection:

    Cloud Armor provides Distributed Denial of Service (DDoS) protection to safeguard applications against large-scale attacks aimed at overwhelming resources. It leverages Google's global infrastructure to absorb and mitigate attacks at the network edge before they can reach the application.

  • Web Application Firewall (WAF):

    Cloud Armor integrates a WAF to protect applications from common web application attacks, such as SQL injection, cross-site scripting (XSS), and other OWASP Top 10 threats. Users can define custom rules to block or allow traffic based on specific conditions like IP address, region, or request types.

  • Access Control:

    Cloud Armor allows users to set rules based on various attributes, such as IP address, geography (region-based filtering), and request headers. This enables granular control over who can access the applications and helps ensure that only authorized traffic is allowed.

  • Scalability:

    The service is designed to scale automatically with your application‚Äôs needs. As the application grows or faces traffic spikes, Cloud Armor ensures that the level of protection adapts without requiring manual intervention.

  • Integration with Google Cloud Load Balancer:

    Cloud Armor works seamlessly with Google Cloud's HTTP(S) Load Balancer to protect both global and regional applications deployed in GCP. This integration enables traffic to be filtered before it reaches the backend servers, ensuring that malicious traffic does not compromise the application.

  • Logging and Monitoring:

    Cloud Armor provides detailed logs that track incoming requests, which help with security auditing, compliance, and troubleshooting. Integration with Google Cloud's operations suite (formerly Stackdriver) allows for real-time monitoring and alerting on suspicious activity.

Cloud Router is a managed service in Google Cloud Platform (GCP) that automatically exchanges route information between your Virtual Private Cloud (VPC) network and your on-premises network using dynamic routing protocols like BGP (Border Gateway Protocol).

Key Functions of Cloud Router:

  • Dynamic Routing:

    Automatically learns and shares route information with on-premises networks through BGP, ensuring that your network stays updated without manual configuration.

  • High Availability:

    Cloud Router supports redundant configurations to maintain connectivity even during failures, ensuring reliable hybrid and multi-cloud environments.

  • Scaling:

    It scales automatically to handle the growth of your network without needing manual intervention or hardware upgrades.

  • Integration with VPN and Interconnect:

    Cloud Router works with Cloud VPN and Dedicated Interconnect to manage hybrid connections dynamically and securely.

  • Reduced Manual Effort:

    Eliminates the need to manually update routes when your network grows or changes, reducing the risk of errors and operational burden.

VPN Tunnels

A VPN Tunnel securely connects your on-premises network or another cloud environment to your Google Cloud VPC over the public Internet using encrypted communication.

  • Encrypts traffic between sites securely.
  • Can be Static (manual routes) or Dynamic (with Cloud Router and BGP).
  • Commonly used for hybrid cloud or multi-cloud networking.

How They Work Together

  • Without Cloud Router: VPN uses static routes; manual updates needed.
  • With Cloud Router: VPN tunnels use dynamic routing (BGP), automatic route exchange.
  • Improves network flexibility and simplifies management.

What is a Firewall Rule in GCP?

A firewall rule in Google Cloud Platform (GCP) is a set of conditions used to control the inbound and outbound traffic to and from Virtual Machine (VM) instances within a VPC (Virtual Private Cloud) network.

Key Points:

  • Traffic Control: Firewall rules control which traffic is allowed or denied to VM instances based on defined parameters like IP address, port, and protocol.
  • Direction: Rules apply to either incoming traffic (ingress) or outgoing traffic (egress).
  • Scope: Can be applied to specific instances or all instances in a network.
  • Evaluation: GCP evaluates the rules top-down; the first matching rule is applied.

Common Components of Firewall Rules:

  • Source/Destination IP: Defines the source or destination IP addresses allowed for traffic.
  • Ports and Protocols: Specifies which ports (e.g., 80, 443) and protocols (e.g., TCP, UDP) are allowed.
  • Action: Can be allow or deny traffic matching the rule.
  • Priority: Each rule has a priority number; lower numbers have higher priority.

Example:

A firewall rule that allows HTTP traffic (port 80) from any source (0.0.0.0/0) to your VMs would look like this:

  • Source IP Range: 0.0.0.0/0 (any IP)
  • Allowed Protocol/Port: TCP port 80
  • Action: Allow

How It Works:

When a request is made to a VM instance, the firewall rule is evaluated based on the source and destination IP addresses, protocol, and port. If the rule matches and allows the traffic, it proceeds to the VM. Otherwise, the traffic is blocked.

Note: This control is essential for securing cloud-based resources and ensuring that only authorized traffic reaches the VM instances.

SaaS
SaaS (Software as a Service) is a cloud computing model where software applications are delivered over the internet. Instead of installing and maintaining software locally, users access it through a web browser. The provider hosts, maintains, updates, and secures the application.
Examples: Gmail, Google Docs, Salesforce, Dropbox.
Deployment Model Description
Public SaaS A single version of the application is used by multiple customers (multi-tenant). Hosted and managed by the SaaS provider. Example: Gmail.
Private SaaS A dedicated instance of the application for a single organization. Offers more control and customization. Example: SaaS apps deployed in a company's private cloud.
Hybrid SaaS Combines public and private SaaS features. Some components run in a public cloud, while sensitive data or services are handled privately.

Stackdriver

Stackdriver (now called Google Cloud Operations Suite) provides monitoring, logging, and diagnostics for applications running on Google Cloud and other environments.

How to Use Stackdriver

  • Monitoring:
    • Tracks performance of VMs, databases, applications, and more.
    • Create dashboards to visualize metrics.
    • Set up alerts to get notified when resources cross thresholds (e.g., CPU usage above 80%).
  • Logging:
    • Collects and stores logs from GCP services and VMs.
    • Use Logs Explorer to search, filter, and analyze logs.
    • Create log-based metrics or alerts directly from logs.

Steps to Set Up Stackdriver

  1. Go to the GCP Monitoring Console.
  2. Enable Monitoring and Logging APIs.
  3. Install Stackdriver agents (optional for detailed VM monitoring).
  4. Create alert policies and custom dashboards as needed.

Benefits

  • Real-time monitoring and troubleshooting.
  • Centralized logging across projects and services.
  • Better uptime with proactive alerts.

VM Types
VM Type Description Use Case Pricing
On-Demand VMs Traditional VMs that are billed for the time they run. No duration limitation. Ideal for workloads that require high availability and cannot be interrupted. More expensive than Preemptible VMs, billed for active usage.
Preemptible VMs Short-lived VMs that can be terminated by Google Cloud if needed. Perfect for fault-tolerant workloads like batch processing and big data analysis. Cheaper, but can be interrupted at any time with a 30-second warning.
Spot VMs Similar to Preemptible VMs but may offer more flexible termination policies. Useful for highly fault-tolerant workloads, like data processing jobs. Cheaper than On-Demand VMs, but can be interrupted at any time.
Committed Use VMs VMs offered at a discount in exchange for a commitment to use them for 1 or 3 years. Suitable for long-term, predictable workloads. Discounted compared to On-Demand VMs, but requires a long-term commitment.
Reserved VMs Varying levels of resource reservation for longer-term use, similar to Committed Use VMs. Best for production environments with a predictable resource need. Discounted compared to On-Demand VMs, with resource reservation.
N1, N2, E2, C2 (VM Families) VMs optimized for different workloads such as general-purpose, compute-intensive, budget, and high-performance computing. Depending on your needs for CPU, memory, and network resources, you can choose the appropriate VM family. Varies depending on the family, offering different price points and performance options.

üõ°Ô∏è Proactive Security Practices

Goal: Prevent incidents before they occur.

  • IAM Least Privilege: Only grant the minimum necessary permissions (e.g., use Storage Viewer instead of Editor).
  • Continuous scanning with SCC: Detect misconfigurations like open firewalls or public buckets before they‚Äôre exploited.
  • VPC Service Controls: Restrict data movement to reduce the risk of data exfiltration.
  • Binary Authorization (for GKE): Enforce signed and verified container images only.
  • Organization Policies: Prevent creation of insecure resources (e.g., public buckets, unmanaged disks).
  • Vulnerability scanning: Scan container images for CVEs before deployment using Container Analysis.
  • CI/CD security gates: Block code with vulnerabilities or policy violations from reaching production.

üö® Reactive Security Practices

Goal: Detect and respond to incidents when they occur.

  • SCC finding triggers automation: Use Pub/Sub and Cloud Functions to isolate VMs or remove risky firewall rules automatically.
  • Incident response runbooks: Predefined steps for responding to security events (e.g., disable access, notify teams, collect logs).
  • Event Threat Detection: Real-time detection of malware, port scanning, and other anomalies in GCP logs.
  • Log analysis and forensics: Use tools like Chronicle or BigQuery to investigate suspicious activity after an incident.
  • Audit logs and Access Transparency: Track who did what, when, and where inside your environment.
  • Backup and disaster recovery: Recover compromised or lost data (e.g., after a ransomware attack).
