JWT (JSON Web Token)

JWT is used to securely transmit information between two parties as a JSON object

Step Action Authentication Factor
1 User logs in by providing username and password Initial Authentication Factor: Username and Password
2 The server verifies the user credentials and generates a JWT Authentication completed using JWT (token generated)
3 User stores JWT (often in local storage or cookies) JWT is stored in the user's browser or app
4 User sends the JWT with each request to authenticate the user on the backend JWT is used to authenticate on each request
5 The server verifies the JWT (checks signature, expiration, etc.) JWT verification performed by the server
6 Access granted if the JWT is valid Access granted based on valid JWT

Common Use Cases for JWT

Use Case Description
Authentication JWT is commonly used to authenticate users in web and mobile applications after they log in, maintaining their session.
Authorization JWT can be used to grant access to resources, ensuring that users have the appropriate permissions for actions or pages.
Single Sign-On (SSO) JWT allows for secure, decentralized Single Sign-On (SSO) by verifying the user’s identity across different services or applications.
API Authentication JWTs are often used to authenticate API requests between services or between client and server.

JWT Structure

Part Description
Header Contains metadata about the token, such as the type of token (JWT) and the signing algorithm (e.g., HMAC, RSA, or SHA256).
Payload Contains the claims or data being transmitted, such as user information, roles, or permissions. Claims can be registered (e.g., 'sub'), public, or private.
Signature Ensures the token has not been tampered with. It is created by signing the header and payload with a secret key or a private key in the case of asymmetric encryption.


Benefits of Using JWT

Benefit Description
Stateless Authentication JWT allows for stateless authentication, meaning the server doesn’t need to store session data. The token itself contains all necessary information.
Scalable Since the server doesn’t need to manage sessions, JWT is scalable and is ideal for large applications or microservices.
Cross-platform JWT can be used across different platforms, making it suitable for web, mobile, and API communication.
Decentralized JWT can be used for Single Sign-On (SSO) and federated identity, enabling authentication across various applications or domains.

Common Security Risks with JWT

Risk Description
Token Expiry If the expiration date of a JWT is too far in the future, it increases the window of attack. Short-lived tokens are recommended.
Insecure Storage Storing JWTs insecurely (e.g., in local storage without proper protection) can make them vulnerable to attacks like Cross-Site Scripting (XSS).
Algorithm Vulnerabilities Weak or improperly configured signing algorithms (e.g., using "none" or weak algorithms like MD5) can allow attackers to forge JWTs.
Token Hijacking If JWTs are intercepted or stolen (via MITM or other attacks), attackers can impersonate users until the token expires.
