Step Action Authentication Factor
1 User attempts to access a resource (Service Provider) using a third-party application (Client) Initial Authentication (via third-party app)
2 The client application sends an OAuth request for authorization to the Authorization Server Authorization Request
3 User is prompted to authenticate and authorize the application to access their resources (e.g., via login) User credentials validated by Authorization Server
4 If authenticated, the Authorization Server sends an authorization code back to the client application Authorization Code
5 The client application exchanges the authorization code for an access token from the Authorization Server Access Token
6 The client application uses the access token to access the user's protected resources from the Resource Server Access with Token

Common Use Cases for OAuth

Use Case Description
Third-Party Login OAuth is commonly used in "Login with Facebook/Google" scenarios, where users can authenticate using their existing account credentials from a trusted third-party.
Delegated Access OAuth is used to grant a third-party application limited access to a user's resources (e.g., access to Google Calendar or Facebook posts) without sharing the user's password.
Mobile App Authentication OAuth is used to authenticate mobile apps that access a user's data without requiring them to input their password directly into the app.
Cloud Service Authentication OAuth is used for secure, token-based authentication to cloud services like Google Cloud, Microsoft Azure, and AWS, allowing secure API calls to access resources.

OAuth Flow and Components

Component Description
Authorization Server The server that authenticates the user and issues tokens (authorization code, access token).
Resource Server The server that holds the protected resources and accepts access tokens to authorize access to them.
Client Application The third-party app that requests access to the user's protected resources.
Authorization Code A temporary code issued by the authorization server after the user authorizes access to their resources, exchanged for an access token.
Access Token A token issued by the authorization server that grants access to the protected resources for a specified period.
Refresh Token A token used to obtain a new access token after the original one expires, without requiring the user to re-authenticate.

Benefits of OAuth

Benefit Description
Delegated Access OAuth enables third-party apps to access user data without requiring the user to share their credentials.
Security OAuth reduces security risks by allowing users to authorize access without exposing their passwords.
Granular Access OAuth allows users to grant limited access to their data, enhancing privacy and security (e.g., allowing an app to only access calendar events, not email).
Single Sign-On (SSO) OAuth can be part of an SSO solution, allowing users to authenticate once and access multiple services without needing to log in each time.

Common OAuth Security Risks

Risk Description
Token Theft If an attacker intercepts or steals an access token, they may impersonate the user and access protected resources.
Authorization Code Interception If the authorization code is intercepted during transmission, an attacker can exchange it for an access token.
Insecure Redirect URIs If OAuth redirect URIs are misconfigured, attackers may redirect the user to a malicious server to steal tokens or credentials.
Insufficient Token Expiry If tokens don't expire or are long-lived, stolen tokens can be used for extended periods before being invalidated.
Cross-Site Request Forgery (CSRF) Malicious websites can make unauthorized requests to a service on behalf of an authenticated user if proper protections are not implemented.
